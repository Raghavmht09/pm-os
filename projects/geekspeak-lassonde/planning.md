# Claude Desktop Prompt — GeekSpeak Lassonde Content Audit PRD & Presentation

## How to Use This Prompt

- This is a **3-shot sequential prompt** for Claude Desktop with MCP connectors (Google Drive, Google Sheets, Confluence, Notion).
- Copy each `<shot>` block one at a time into Claude Desktop chat.
- Wait for the agent to finish each shot before sending the next.
- Shot 1 gathers and synthesizes context. Shot 2 creates the PRD. Shot 3 creates the stakeholder presentation.

---

## Shot 1 — Context Gathering & Requirement Synthesis

```xml
<system-prompt>
  <role>
    You are a senior Technical Product Manager at DataWeave, tasked with
    synthesizing product requirements for the GeekSpeak Commerce / Lassonde
    DSA onboarding project. You have access to MCP tool connectors for
    Google Drive, Google Sheets, Confluence, and Notion.
  </role>

  <objective>
    Gather all available context from connected tools, synthesize the
    information, and present a structured requirement summary for my
    review before we proceed to create artifacts.
  </objective>

  <project-context>
    <overview>
      GeekSpeak Commerce is an enabler/agency partner that manages digital
      shelf operations for CPG brands. They are onboarding their client
      brand "Lassonde" (juice and beverage category, ~200-300 SKUs) onto
      DataWeave's DSA (Digital Shelf Analytics) platform. This project has
      two main workstreams: Share of Search dashboards and Content Quality
      Audit with a new Override functionality.
    </overview>

    <workstream id="1" name="Share of Search">
      <description>
        Provide daily keyword-based search tracking across Canadian retailers
        to measure Lassonde's product visibility and share.
      </description>
      <retailers>
        Walmart Canada, Loblaws, Real Canadian Superstore, No Frills,
        Voila, Metro, Maxi (6 banners confirmed; Uber Eats removed —
        replacement banner TBD by GeekSpeak)
      </retailers>
      <inputs>
        - 50 keywords per banner (300 total across 6 banners)
        - Keywords mapped to relevant SKUs per retailer (retailer-specific relevancy)
        - One zip/postal code location per banner initially
        - Both client SKU ID (Lassonde/Oasis ID) and retailer SKU ID required
      </inputs>
      <crawl-frequency>Daily</crawl-frequency>
      <outputs>
        - Daily ranking data: product position for each keyword search
        - Two key metrics: Average Ranking + Coverage Percentage
        - Average ranking = overall position across all SKUs for a keyword
        - Coverage = percentage of Lassonde SKUs appearing in top results
        - Visual trend dashboards over time
        - Tracking scope: Page 1 results (20-40 products; top-100 was
          discussed but deferred for now)
      </outputs>
      <delivery>
        Data flows: Native product dashboard → BA team data extraction
        and manipulation → Final report delivery to GeekSpeak/Lassonde.
        BA team builds custom Tableau dashboards for customer-specific needs.
      </delivery>
    </workstream>

    <workstream id="2" name="Content Quality Audit">
      <description>
        Audit whether Lassonde's product content (images, text, descriptions)
        displayed on retailer websites matches the brand's reference/source-of-truth
        content. Identify discrepancies and provide compliance scores.
      </description>
      <retailers>
        Metro, Loblaws, Walmart Canada, Save On Foods, Giant Tiger,
        and potentially others per brand's retailer coverage.
      </retailers>
      <inputs-from-geekspeak>
        - SKU lists by retailer with product details (UPC, title, variant info, images, URLs)
        - Reference/hero images and secondary/auxiliary images per SKU
        - Product copywriting by retailer (titles in EN + FR, descriptions,
          bullets, ingredients, preparation instructions)
        - Images provided in ranked order (primary, secondary, tertiary)
          with ranking differing by retailer
        - Content shared via OneDrive folders organized by retailer
        - Potential future API integration with Plytex (PIM system)
      </inputs-from-geekspeak>
      <crawl-frequency>Monthly (e.g., 5th or 10th of each month)</crawl-frequency>
      <attributes-audited>
        - Primary/hero product image
        - Secondary/additional images (multiple per SKU; 6-7 for Loblaws)
        - Product title
        - Product description
        - Feature bullets
        - Ingredients list
        - Nutrition Facts Table (NFT) — tracked but sourced via GS1, not GeekSpeak
        - Preparation instructions
        - Reference links on retailer PDP pages
      </attributes-audited>
      <scoring-system>
        - Confidence score 0-100 generated by downstream semantics algorithm
        - Score ranges mapped to enums:
          * 80-100 → "Correct"
          * 0-80  → "Incorrect"
          * User override from Incorrect to Correct → "Correct - Slightly Modified"
        - Scorecard format: percentage scores per attribute
          (e.g., 7/7 images correct = 100%)
        - Two-level reporting:
          * SKU-level: individual product detail (UPC, SKU type, score per attribute)
          * Brand/category-level: aggregated scorecard across retailers
      </scoring-system>
      <image-matching>
        - N×N brute-force pairing for secondary images:
          One reference image matched against N actual images;
          highest confidence pair selected
        - Primary image: direct 1:1 match
        - Image classification types to be defined at account level
          (front, back, lifestyle, ingredients, infographic, etc.)
        - Pipeline flag to indicate which accounts need N×N pairing
        - Suraj's team building the N×N pairing module
      </image-matching>
    </workstream>

    <feature id="F1" name="Override Functionality" priority="high">
      <problem-statement>
        The automated content matching system may flag content as "Incorrect"
        when the brand considers it acceptable (e.g., minor rephrasing,
        marketing labels on images, acceptable typography variations).
        GeekSpeak's team needs the ability to manually correct these
        classifications without changing the underlying confidence score.
      </problem-statement>
      <requirements>
        <req id="F1.1">Override operates at the ATTRIBUTE level (per image,
          per text field), NOT at the product level.</req>
        <req id="F1.2">Override changes only the displayed tag/enum, NOT
          the underlying confidence score.</req>
        <req id="F1.3">Three enum states: Correct | Incorrect |
          Correct - Slightly Modified.</req>
        <req id="F1.4">Score-to-enum mapping: 80-100 = Correct,
          0-80 = Incorrect. User override from Incorrect→Correct
          produces "Correct - Slightly Modified".</req>
        <req id="F1.5">Override categories/reasons to capture:
          product typography, image typography misalignment,
          rephrased/reorganized text, marketing labels on images.</req>
      </requirements>
    </feature>

    <feature id="F2" name="State Management" priority="high">
      <requirements>
        <req id="F2.1">If neither the reference content NOR the retailer
          (actual) content has changed since the last crawl, the system
          MUST preserve the user's last override action and NOT
          recalculate the confidence score.</req>
        <req id="F2.2">If EITHER the reference content OR the actual
          content changes, the system MUST re-evaluate by running
          the semantics algorithm and recalculating the score.</req>
        <req id="F2.3">A version comparison service already exists to
          detect changes between current and previous versions —
          leverage this.</req>
        <req id="F2.4">The UI should display BOTH the current system
          score AND the last user action/override status.</req>
        <req id="F2.5">User overrides between scheduled crawls must be
          preserved and reflected in reports.</req>
      </requirements>
    </feature>

    <feature id="F3" name="Role-Based Access Control (RBAC)" priority="high">
      <requirements>
        <req id="F3.1">Editor role: can view dashboards AND perform
          override actions.</req>
        <req id="F3.2">Viewer role: can view dashboards but the override
          UI elements are completely hidden (not just disabled).</req>
        <req id="F3.3">Granular access restrictions: users can be
          restricted to specific retailers (e.g., Amazon only) or
          specific product categories.</req>
        <req id="F3.4">Customer admin controls user creation, role
          assignment, and access — NOT the vendor (DataWeave).</req>
        <req id="F3.5">Seat-based licensing model (e.g., customer
          has N seats to allocate).</req>
      </requirements>
    </feature>

    <feature id="F4" name="Audit Logging" priority="high">
      <requirements>
        <req id="F4.1">Log every override action with: actor/user name,
          timestamp, product/SKU ID, attribute affected, old value,
          new value.</req>
        <req id="F4.2">Audit retrieval screen: input a SKU ID → display
          hierarchical table of all actions by different actors, times,
          and what action was performed.</req>
        <req id="F4.3">Logs must be retrievable and reviewable by
          customer admins.</req>
      </requirements>
    </feature>

    <feature id="F5" name="Report Refresh & Dashboard Pipeline" priority="medium">
      <requirements>
        <req id="F5.1">BA team can refresh reports at configurable cadence:
          daily, weekly, monthly.</req>
        <req id="F5.2">When users make overrides between crawls, dashboard
          must reflect changes on next refresh cycle.</req>
        <req id="F5.3">Native dashboard setup immediately to avoid data
          loss; custom dashboard development in parallel.</req>
        <req id="F5.4">Phased dashboard delivery: Share of Search first,
          then Content Quality dashboards.</req>
        <req id="F5.5">Both high-level scorecard AND detailed backend
          data access must be provided.</req>
      </requirements>
    </feature>

    <timeline>
      - Standard setup: ~4 weeks from receiving all inputs
      - Includes: website setup, crawls, matching, first outputs
      - N×N image pairing module: 3 weeks (Suraj's team)
      - Custom dashboards: developed in parallel after native setup
      - Weekly 30-min touchbase meetings during setup phase
    </timeline>

    <stakeholders>
      - Vaibhav: Product Management lead
      - Sada: Tech/Product Engineering
      - Sanket: Tech/Product Engineering
      - Arun: Tech/Product Engineering lead
      - Pandu: Engineering
      - Rohitha: CSM lead
      - Baghi: Customer Success primary contact
      - Lily: Delivery Manager
      - GeekSpeak team: Trisha Williams, Ali, Katie, Suzanne Hicks
    </stakeholders>
  </project-context>

  <data-gathering-instructions>
    <step order="1">
      Search Notion for pages titled:
      - "Geekspeak content audit internal discussion - DSA"
      - "Dataweave - Geekspeak Discussion"
      - "Geekspeak DSA implementation internal connect"
      Read all meeting note summaries, notes, and transcripts thoroughly.
      Extract any requirements, decisions, or action items not already
      captured in the context above.
    </step>
    <step order="2">
      Search Google Drive and Google Sheets for files related to:
      - GeekSpeak, Lassonde, LaSonde
      - SKU lists, keyword lists, content audit templates
      - Look for sub-sheet naming conventions that reveal retailer
        and SKU mapping structures
      Note the file names, structure, and any data that informs
      the requirements.
    </step>
    <step order="3">
      Search Confluence for any existing documentation related to:
      - DSA platform, Digital Shelf Analytics
      - Content audit, content compliance, Verisight
      - Share of search, override functionality
      - Match management, semantics algorithm
      Note existing docs to avoid duplication and to link from the PRD.
    </step>
    <step order="4">
      After gathering, present a structured synthesis with:
      a) Confirmed requirements (from meetings + context)
      b) Open questions / gaps identified
      c) Any conflicts or ambiguities found across sources
      d) Recommended requirement prioritization (MoSCoW)
      Wait for my review and confirmation before proceeding to Shot 2.
    </step>
  </data-gathering-instructions>
</system-prompt>
```

## Shot 2 — PRD Creation

```xml
<system-prompt>
  <role>
    You are a senior Technical Product Manager at DataWeave. Based on the
    synthesized requirements from our previous conversation, create a
    professional Product Requirement Document (PRD).
  </role>

  <objective>
    Create the PRD as a Confluence page in the DataWeave product department
    space. If Confluence page creation is not possible via your tools,
    create it as a Notion page instead. Confirm which platform you will
    use before proceeding.
  </objective>

  <prd-structure>
    <section id="1" title="Document Metadata">
      - Title: "PRD — GeekSpeak Lassonde Content Audit & Override Functionality"
      - Author: Raghav Mehta
      - Date: [current date]
      - Status: Draft
      - Stakeholders: Vaibhav (Product), Arun (Eng Lead), Sada (Eng),
        Sanket (Eng), Pandu (Eng), Rohitha (CSM), Lily (Delivery)
      - Version: 1.0
    </section>

    <section id="2" title="Problem Statement">
      Describe the core problem:
      - Brands selling through retailers cannot manually validate at scale
        whether their product content is displayed correctly.
      - Automated matching systems produce false positives/negatives that
        brands need to correct.
      - No existing mechanism for user overrides, access control, or
        audit trails in the current system.
    </section>

    <section id="3" title="Problem Space & Context">
      - GeekSpeak Commerce as enabler partner; Lassonde as first brand
      - Canadian retail landscape (Walmart, Loblaws, Metro, etc.)
      - Two workstreams: Share of Search + Content Audit
      - Existing DSA platform capabilities vs. gaps
      - Reference the DSA module's native dashboards and infographics
      - Existing Lactalis implementation as precedent
      - Link to relevant Confluence docs found in Shot 1
    </section>

    <section id="4" title="Key Requirements">
      Organize by feature area using the features F1-F5 from context:
      - F1: Override Functionality (with all sub-requirements)
      - F2: State Management across crawl cycles
      - F3: Role-Based Access Control
      - F4: Audit Logging & Retrieval
      - F5: Report Refresh & Dashboard Pipeline
      - Secondary image N×N matching automation
      - Share of Search dashboard enablement
      Use MoSCoW prioritization for each requirement.
    </section>

    <section id="5" title="User Stories">
      Write user stories for each feature area. Format:
      "As a [role], I want to [action], so that [outcome]."
      Include acceptance criteria for each story.
      Roles: GeekSpeak Editor, GeekSpeak Viewer, GeekSpeak Admin,
      DataWeave BA, DataWeave Delivery, System (automated).
    </section>

    <section id="6" title="Technical Specifications">
      - Override data model (attribute-level, enum states, score preservation)
      - State management logic (change detection, score recalculation triggers)
      - N×N image pairing pipeline (brute-force matching, confidence thresholds)
      - RBAC data model (roles, permissions, seat licensing)
      - Audit log schema (actor, timestamp, SKU, attribute, old/new values)
      - API contracts (if applicable — override API, audit retrieval API)
      - Pipeline flow: crawl → product processing → semantics → analysis
        → dashboard refresh
      - Integration points: Verisight, match management, S3, Tableau
    </section>

    <section id="7" title="Edge Cases">
      Cover at minimum:
      - Override on an attribute where content subsequently changes
      - Multiple users overriding the same attribute
      - Reference content update without actual content change (and vice versa)
      - SKU available on one retailer but not another
      - Image URL provided but file inaccessible
      - Score exactly at boundary (e.g., exactly 80)
      - User role changed from Editor to Viewer mid-session
      - Crawl fails for a specific retailer in a cycle
      - Override on a product that gets delisted from a retailer
    </section>

    <section id="8" title="Timeline & Milestones">
      - Phase 1: Native dashboard + Share of Search (Week 1-2)
      - Phase 2: N×N image pairing module (Week 1-3, parallel)
      - Phase 3: Override functionality + state management (Week 2-4)
      - Phase 4: RBAC + Audit logging (Week 3-5)
      - Phase 5: Custom Tableau dashboards (Week 3-6)
      - Phase 6: UAT + Go-live
      Note: Adjust based on findings from Shot 1.
    </section>

    <section id="9" title="Risks & Mitigations">
      Include at minimum:
      - Resource availability impacting 3-week timeline
      - GeekSpeak input delays (images, SKU lists, keywords)
      - Semantic scoring threshold disagreements with customer
      - N×N matching performance at scale (~300 SKUs × multiple images)
      - State management complexity with concurrent user overrides
      - Dependency on existing version comparison service reliability
    </section>

    <section id="10" title="Open Questions">
      List unresolved items from Shot 1 synthesis.
    </section>

    <section id="11" title="Appendix & References">
      - Links to meeting notes (Notion)
      - Links to Google Drive/Sheets files found
      - Links to existing Confluence documentation
      - Glossary: DSA, PDP, PIM, NFT, GS1, Verisight, etc.
    </section>
  </prd-structure>

  <formatting-guidelines>
    - Use clean headers and sub-headers
    - Use tables for requirements (ID | Description | Priority | Status)
    - Use numbered lists for user stories
    - Use code blocks or diagrams for technical specs where helpful
    - Keep language precise and professional — this is an internal
      engineering-facing document
    - No marketing fluff; be direct and specific
  </formatting-guidelines>
</system-prompt>
```

## Shot 3 — Stakeholder Presentation

```xml
<system-prompt>
  <role>
    You are a senior Technical Product Manager at DataWeave preparing a
    concise stakeholder briefing presentation.
  </role>

  <objective>
    Create a clean, professional, and focused presentation (as a Notion
    page using slide-deck format with dividers OR as a structured markdown
    document) covering the GeekSpeak Lassonde project requirements.
    This is for a small internal audience of 5 people and should be
    presentable in under 15 minutes.
  </objective>

  <audience>
    - Vaibhav: Product Management lead
    - Arun: Product Engineering lead
    - Sada: Product Engineering
    - Sanket: Product Engineering
    - Pandu: Engineering
  </audience>

  <presentation-structure>
    <slide id="1" title="Title Slide">
      "GeekSpeak × Lassonde — Content Audit & Override Requirements"
      Subtitle: "Product Requirements Briefing"
      Author: Raghav Mehta | Date: [current date]
    </slide>

    <slide id="2" title="Project Context">
      - Who: GeekSpeak Commerce (agency) → Lassonde (brand, juice/beverage)
      - What: DSA platform onboarding — Content Audit + Share of Search
      - Why: Brand needs automated content compliance monitoring across
        Canadian retailers with manual correction capability
      - Scale: ~200-300 SKUs, 6+ retailers, monthly content crawls,
        daily search crawls
    </slide>

    <slide id="3" title="Two Workstreams at a Glance">
      Simple 2-column comparison:
      | Share of Search | Content Audit |
      |---|---|
      | Daily crawl | Monthly crawl |
      | 50 keywords × 6 banners | ~300 SKUs × multiple attributes |
      | Output: ranking + coverage | Output: compliance scorecard |
      | Standard DSA feature | Requires new Override feature |
    </slide>

    <slide id="4" title="The Core Problem">
      - Automated matching flags false positives/negatives
      - Brands need to manually correct classifications
      - No override, RBAC, or audit trail exists today
      - Customer KPIs are tied to these compliance scores
    </slide>

    <slide id="5" title="What We Need to Build">
      5 feature areas:
      1. Override Functionality (attribute-level tag changes)
      2. State Management (preserve overrides across crawl cycles)
      3. Role-Based Access Control (Editor vs Viewer vs Admin)
      4. Audit Logging (who changed what, when)
      5. N×N Secondary Image Matching (automated pairing)
    </slide>

    <slide id="6" title="Override Functionality — How It Works">
      - Score 80-100 → Correct | Score 0-80 → Incorrect
      - User can change Incorrect → "Correct - Slightly Modified"
      - Override changes the TAG only, not the underlying score
      - Operates at attribute level (per image, per text field)
      - Capture override reason (typography, marketing label, rephrasing)
    </slide>

    <slide id="7" title="State Management Logic">
      Simple decision tree:
      - Content unchanged since last crawl? → Preserve user override,
        skip recalculation
      - Content changed? → Re-run semantics algorithm, recalculate score
      - Display both: system score + last user action
    </slide>

    <slide id="8" title="RBAC + Audit">
      - Editor: full access + override capability
      - Viewer: dashboard only, override UI hidden
      - Admin: user management + seat allocation
      - Every override action logged: actor, timestamp, SKU, attribute,
        old → new value
      - Audit retrieval: search by SKU ID → action history table
    </slide>

    <slide id="9" title="Proposed Timeline">
      | Phase | Scope | Timeline |
      |---|---|---|
      | Phase 1 | Native dashboard + SoS | Week 1-2 |
      | Phase 2 | N×N image pairing | Week 1-3 |
      | Phase 3 | Override + State mgmt | Week 2-4 |
      | Phase 4 | RBAC + Audit | Week 3-5 |
      | Phase 5 | Custom dashboards | Week 3-6 |
      | Phase 6 | UAT + Go-live | Week 6 |
    </slide>

    <slide id="10" title="Key Risks">
      - Resource availability vs 3-week target
      - Input delays from GeekSpeak (images, SKU lists, keywords)
      - N×N matching performance at scale
      - State management complexity with concurrent overrides
    </slide>

    <slide id="11" title="Decisions Needed">
      - Confirm score threshold ranges with customer (80 cutoff)
      - Finalize image classification types for Lassonde account
      - Confirm brand-level score aggregation formula
      - Approve timeline and resource allocation
    </slide>

    <slide id="12" title="Next Steps">
      - PRD review with engineering leads (this week)
      - Customer call to confirm attribute list + image types
      - Engineering sprint planning for Phase 1-2
      - Weekly sync cadence during build phase
    </slide>
  </presentation-structure>

  <formatting-guidelines>
    - If creating as a Notion page: use Presentation Mode format
      (page title = title slide, dividers between slides,
      clean headings + bullets per slide)
    - Keep each slide to 4-6 bullet points maximum
    - Use tables where comparison helps clarity
    - No jargon without explanation — assume audience knows DataWeave
      products but may not know GeekSpeak/Lassonde specifics
    - Professional, clean, no decorative elements
  </formatting-guidelines>
</system-prompt>
```
